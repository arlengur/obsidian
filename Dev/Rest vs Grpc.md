Архитектурный стиль взаимодействия компонентов распределенного приложения в сети.
Это набор правил и ограничений, которому должно удовлетворять приложение.

Свойства Rest:
- Модель client-server. 
	- Отделение клиентской части от серверной позволяет облегчить масштабируемость и переносимость приложения.
- Отсутствие состояния (Stateless)
	- В запросе содержится вся информация для его обработки, сервер не хранит состояние
- Кэширование 
	- Клиент может что-то кэшировать, на сервере кэша нет
- Единообразие интерфейса (Uniform interface) 
	- Определенный набор запросов: get, post, put, patch, delete
- Слои (Layered System) 
	- Клиент не имеет возможности определить взаимодействует ли он напрямую с сервером или через промежуточные слои. 
	- Применение промежуточных слоев поможет сбалансировать нагрузку между серверами, обеспечить дополнительную безопасность, помочь организовать кэширование ответов
	- Примеры: кэш, сериализация-десериализация бизнес-логики

| Протокол | Бинарный | Текстовый |

| Мультиплексирование | + | - |
| Запросы в соединении | Много | 1 |

Мультиплексирование - способность отдавать сообщение по частям

Protobuf - механизм для сериализации данных

```proto
message Person {
  required string name = 1;
  required int32 id = 2;
  optional string email = 3;
}
```

rpc - удаленный вызов процедур
