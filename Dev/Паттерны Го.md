
# Паттерн Pipeline (Конвейер) 

Структурный паттерн конкурентности, который позволяет разбить сложный процесс обработки данных на последовательность независимых этапов (stages). Каждый этап представляет собой горутину, которая получает данные из входящего канала, обрабатывает их и отправляет результат в исходящий канал. 

Основные компоненты Pipeline
- Stage (Этап): Функция, выполняющая конкретную задачу, запущенная в горутине.
- Channel (Канал): Связывает этапы, передавая данные от одного к другому.
- Generator (Источник/Producer): Специальный первый этап, преобразующий данные в поток.
- Consumer (Приемник): Последний этап, который считывает итоговые данные.

```go
// 1. Stage: Генератор данных (Producer)
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out) // Обязательно закрываем канал
    }()
    return out
}

// 2. Stage: Обработка данных (например, возведение в квадрат)
func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out) // Обязательно закрываем канал
    }()
    return out
}

func main() {
    // Подключаем этапы конвейера
    data := generate(1, 2, 3, 4, 5) // Этап 1
    results := square(data)        // Этап 2

    // 3. Stage: Потребление данных (Consumer)
    for res := range results {
        fmt.Println(res)
    }
}
```

Преимущества Pipeline в Go
- Конкурентность: Все этапы могут работать параллельно, используя несколько ядер CPU.
- Потоковая обработка: Данные обрабатываются по мере поступления, не нужно ждать загрузки всего объема.
- Гибкость: Этапы можно легко добавлять, удалять или менять местами.
- Изоляция: Каждый этап работает в своей горутине, что упрощает масштабирование. 

Важные принципы реализации
- Закрытие каналов: Каждый этап обязан закрывать исходящий канал, когда заканчивает работу (close(out)). Это сигнал следующему этапу завершить цикл range.
- Защита от утечек: Если этап не считывает все данные из канала, горутина-отправитель зависнет (deadlock).
- Управление через Context: Для корректной остановки всего конвейера, если один из этапов завершился с ошибкой, следует использовать context.Context

Fan-out, Fan-in (Продвинутый Pipeline)
Если один из этапов конвейера работает медленно (например, запрос к базе данных), его можно распараллелить (Fan-out), запустив несколько экземпляров этой горутины, а затем собрать результаты обратно (Fan-in).
