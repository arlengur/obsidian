# Семафор

Семафор — синхронизация доступа к общему ресурсу, представляющий собой счётчик, над которым можно провести две атомичные операции: увеличение и уменьшение. Если значение счётчика уже равно нулю, операция уменьшения должна блокировать до тех пор, пока счётчик не будет увеличен.

Самую простую реализацию семафора можно сделать через канал
```go
	// Инициализируем канальный семафор, позволяющий реагировать на отсутствие записи
	semaChan := make(chan struct{}, 10)
	for range 10 {
		semaChan <- struct{}{}
	}

	// Создаём тестовый сервер для ответа на запросы
	srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		select {
		case <-semaChan: // пытаемся взять ресурс из канала
		default:
			w.WriteHeader(http.StatusTooManyRequests)
			return
		}
		time.Sleep(200 * time.Millisecond) // имитируем работу
		w.WriteHeader(http.StatusOK)
		semaChan <- struct{}{}
	}))

	wg := sync.WaitGroup{} // создаём двадцать горутин, которые одновременно попытаются вызвать метод на сервере
	wg.Add(20)
	for i := range 20 {
		go func() {
			resp, _ := srv.Client().Get(srv.URL)
			if resp.StatusCode == http.StatusTooManyRequests {
				fmt.Printf("%d: ratelimited\n", i)
			}
			wg.Done()
		}()
	}
	wg.Wait()
```

В результате ровно десять из двадцати горутин не смогут выполнить запрос.

Если нужно гарантированно блокировать исполнение при полной занятости ресурса — можно не инициализировать канал пустыми структурами, а просто поменять местами чтение из канала и запись
```go
// Инициализируем канальный семафор
	semaChan := make(chan struct{}, 10)

	// Создаём тестовый сервер для ответа на запросы
	srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		semaChan <- struct{}{}
		time.Sleep(200 * time.Millisecond) // имитируем работу
		w.WriteHeader(http.StatusOK)
		<-semaChan
	}))
	//...
```

Но в случаях, когда нужно обрабатывать контексты и занимать разное количество ресурсов в зависимости от нагрузки, лучше использовать готовую реализацию семафора из внешнего пакета golang.org/x/sync/semaphore
- NewWeighted (n int64) - создаёт семафор с количеством ресурсов n.  
- Acquire (ctx context. Context, n int64) error — пытается получить от семафора n ресурсов, блокируя до успешности или отмены контекста (например, при отмене или истечении времени). Возвращает nil при успешном запросе ресурсов или ошибку контекста
- Release (n int64) — вовращает в семафор n ресурсов. Если количество возвращаемых ресурсов больше занятых — функция запаникует
- TryAcquire (n int64) bool — пытается получить от семафора n ресурсов, не блокируя. Возвращает true при успешном получении, false при неуспешном

В качестве примера использования сделаем небольшую программу, которая ограничивает количество одновременных запросов
```go
	sema := semaphore.NewWeighted(10) // Создаём семафор

	// Создаём тестовый сервер для ответа на запросы
	srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		if ok := sema.TryAcquire(1); !ok { // пытаемся получить ресурс на обработку одного запроса
			w.WriteHeader(http.StatusTooManyRequests)
			return
		}
		time.Sleep(200 * time.Millisecond) // имитируем работу
		w.WriteHeader(http.StatusOK)
		sema.Release(1) // отпускаем ресурс
	}))

	wg := sync.WaitGroup{} // создаём двадцать горутин, которые одновременно попытаются вызвать метод на сервере
	wg.Add(20)
	for i := range 20 {
		go func() {
			resp, _ := srv.Client().Get(srv.URL)
			if resp.StatusCode == http.StatusTooManyRequests {
				fmt.Printf("%d: ratelimited\n", i)
			}
			wg.Done()
		}()
	}
	wg.Wait()
```

# Worker pool

Создадим три горутины, которые получают работу через первый канал и отдают результат через второй канал. Таким образом можно параллелизовать задачи на ограниченное количество рабочих горутин.
```go
func main() {
	const numJobs = 5
	jobs := make(chan int, numJobs)
	results := make(chan int, numJobs)

	// Инициализация рабочих горутин
	for w := 1; w <= 3; w++ {
		go worker(w, jobs, results)
	}

	// Отправка работы в горутины
	for j := 1; j <= numJobs; j++ {
		jobs <- j
	}
	close(jobs) // подача сигнала о завершении работы

	// Сбор результатов
	for a := 1; a <= numJobs; a++ {
		<-results
	}
}

func worker(id int, jobs <-chan int, results chan<- int) {
	for j := range jobs { // работает до тех пор, пока канал jobs открыт
		fmt.Println("worker", id, "started  job", j)
		time.Sleep(500 * time.Millisecond) // имитация работы
		fmt.Println("worker", id, "finished job", j)
		results <- j * 2
	}
}
```

Рассмотрим альтернативный вариант, использующий канал не для распредления работы между горутинами, а для хранения самих рабочих объектов

```go
type worker struct {
	id          int
	accumulator int
}

func (w *worker) accumulate(input int) (output int) {
	time.Sleep(time.Second) // имитация работы
	w.accumulator += input
	return output
}

func main() {
	const maxWorkers = 5
	const totalJobs = 10

	// Инициализация пула объектов
	workerPool := make(chan *worker, maxWorkers)
	for i := range maxWorkers {
		workerPool <- &worker{id: i}
	}

	wg := sync.WaitGroup{}
	for range totalJobs {
		worker := <-workerPool
		wg.Add(1)
		go func() {
			worker.accumulate(1)
			workerPool <- worker
			wg.Done()
		}()
	}

	// Заверешение работы и вывод результатов
	wg.Wait()
	close(workerPool)
	for w := range workerPool {
		fmt.Printf("worker %d: %d\n", w.id, w.accumulator)
	}
}
```

В результате работа распределится поровну между рабочими объектами. В данном примере канал, по сути, используется как циклическая FIFO-очередь для рабочих объектов.

Такой подход даёт больше контроля над исполнением задач.  
Его можно использовать, когда работу нельзя просто случайным образом распределить между горутинами. Также такой подход можно использовать и для переиспользования данных (например, буферов).

# Fan-out

Бывают случаи, когда необходимо разослать одни и те же данные сразу нескольким горутинам (fan-out). 

```go
func main() {
	maxWorkers := 5

	// Инициализация рабочих горутин
	workers := make([]chan int, 5)
	for i := range maxWorkers {
		workers[i] = worker()
	}

	// Объединение каналов
	inChan := fanOut(workers...)

	inChan <- 20
	close(inChan)

	// Как корректно дождаться отрабатывания всех горутин — задача на дом :)
	time.Sleep(time.Second)
}

// Объединяющая каналы функция
func fanOut(outChannels ...(chan int)) (inChan chan int) {
	inChan = make(chan int)
	go func() {
		// Каждое сообщение из inChan копируется в каналы рабочих горутин
		for input := range inChan {
			for _, ch := range outChannels {
				ch <- input
			}
		}
		// Закрытие каналов рабочих горутин после закрытия inChan
		for _, ch := range outChannels {
			close(ch)
		}
	}()
	return
}

func worker() (inChan chan int) {
	inChan = make(chan int)
	go func() {
		for d := range inChan {
			fmt.Printf("received %d\n", d)
		}
		fmt.Println("closing")
	}()
	return inChan
}
```

# Fan-in

Канал для чтения данных из нескольких горутин

```go
func main() {
	maxWorkers := 5

	// Инициализация рабочих горутин
	workers := make([]chan int, 5)
	for i := range maxWorkers {
		workers[i] = make(chan int)
		go worker(workers[i])
	}

	// Объединение каналов
	outChan := fanIn(workers...)

	for d := range outChan {
		fmt.Printf("received %d\n", d)
	}
}

// Объединяющая каналы функция
func fanIn(inChannels ...(chan int)) (outChan chan int) {
	outChan = make(chan int, len(inChannels))

	// Необходимо для ожидания выхода читающих горутин
	wg := sync.WaitGroup{}
	wg.Add(len(inChannels))
	// Для каждого входного канала создаём горутину
	for _, ch := range inChannels {
		go func(ch chan int) {
			for d := range ch {
				outChan <- d
			}
			wg.Done()
		}(ch)
	}
	// Создаём горутину, которая закроет канал вывода после выхода всех
	// горутин, читающих из входного канала
	go func() {
		wg.Wait()
		close(outChan)
	}()
	return
}

func worker(outChan chan int) {
	outChan <- rand.Intn(1000)
	close(outChan)
}
```
