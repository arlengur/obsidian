
Изначально - это протокол для передачи текста. Сейчас используется для взаимодействия распределенных систем.

Построен на архитектуре клиент-сервер.

- Клиенты (client). Инициируют соединение, отправляют запрос и ожидают получения ответа. 
- Серверы (server). Ожидают запросы со стороны клиентов, производят необходимые для обработки запроса действия и возвращают клиенту ответ
- Прокси. Обрабатывают запросы от клиентов и пересылают серверу.
	- Кеширующие прокси обрабатывают запрос от клиента и отдают ответ, если он есть в кеше
	- Авторизирующие прокси. Проверяют доступность выполнения запроса.

Основным объектом работы в HTTP является URI (Uniform Rescource Identifier — унифицированный идентификатор ресурса). Ресурс может быть физическим (файл на диске) или абстрактным (сущность в приложении). Пример: https://example.com (web), mailto:user@example.com (email), tel:+1-816-555-1212 (phone)

# Понятия
- Заголовки — специальная метаинформация, передаваемая как часть запроса и часть ответа. Благодаря им можно передавать даже двоичные данные.
- Методы — это по сути типы сообщений HTTP. Чувствительны к регистру. Нет списка обязательных методов, которые должен реализовать сервер или клиент. Если сервер не реализует какой-то метод, то возвращает клиенту сообщение с кодом 501 Not implemented, не реализовано со стороны сервера. Если сервер не поддерживает метод именно для указанного ресурса, то возвращает сообщение с кодом 405, not allowed, метод не разрешен.
- Код состояния (Status Code) — трехзначное целое число, обязательно присутствует в ответе сервера и кратко описывает результат запроса. 
- Тело сообщения — собственно контент сообщения, то есть то, что мы хотим передать серверу или получить от сервера

# Методы Http
- GET Используется для запроса содержимого указанного ресурса. С помощью метода GET можно также начать какой-либо процесс. В этом случае в тело ответного сообщения следует включить информацию о ходе выполнения процесса. Параметры запроса передаются в URI с помощью "?". Запросы должны быть идемпотентными (при выполнении запроса с одними и теми же параметрами будет возвращен один и тот же результат), это позволяет кешировать результаты.
- POST Применяется для передачи пользовательских данных заданному ресурсу. Передаваемые данные (в примере с блогами — текст комментария) включаются в тело запроса. Аналогично с помощью метода POST обычно загружаются файлы на сервер. Не идемпотентный, результат ответа не кешируется.
- HEAD Аналогичен методу GET, за исключением того, что в ответе сервера отсутствует тело. 
- PUT Применяется для загрузки содержимого запроса на указанный в запросе URI. Если по заданному URI не существует ресурса, то сервер создаёт его. 
- PATCH Аналогично PUT, применяется для частичного изменения ресурса, в теле ресурса можно передать не весь ресурс а изменяемую часть.
- DELETE Удаляет указанный ресурс.

# Коды состояний HTTP

| Код | Класс                            | Назначение                                                                                                       |
| --- | -------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| 1xx | Информационный (Informational)   | Информирование о процессе передачи                                                                               |
| 2xx | Успех (Success)                  | Информирование о случаях успешного принятия и обработки ответа                                                   |
| 3xx | Перенаправление (Redirection)    | Сообщает клиенту, что для успешного выполнения операции необходимо сделать другой запрос (обычно по другому URI) |
| 4xx | Клиентская ошибка (Client Error) | Указывает на ошибку со стороны клиента.                                                                          |
| 5xx | Ошибка сервера (Server error)    | Информирует о случаях неуспешного выполения операции по вине сервера                                             |
## HTTP/1.1 и HTTP/2

- Формат данных: HTTP/1.1 передает данные в текстовом виде, что медленнее. HTTP/2 использует бинарное (двоичное) кадрирование, что делает обработку более эффективной.
- Мультиплексирование: HTTP/1.1 требует ожидания ответа на предыдущий запрос (блокировка начала очереди). HTTP/2 позволяет отправлять множество запросов одновременно через одно соединение.
- Сжатие заголовков: HTTP/2 использует алгоритм HPACK, значительно уменьшая размер заголовков и ускоряя передачу данных.
- Серверные пуши (Server Push): HTTP/2 позволяет серверу отправлять ресурсы (например, CSS или JS) клиенту до того, как тот их запросит, ускоряя рендеринг страницы.
- Производительность: HTTP/2 эффективнее использует сетевые ресурсы, что критично для современных тяжелых сайтов. 

HTTP/2 полностью сохраняет структуру HTTP (методы, URL, коды ответов), но кардинально меняет способ транспортировки данных.
